# 2026-06-13 (3) — Neovim 0.12.2 compatibility fixes (nvim-tree keymaps + treesitter)

The morning's "verified booting clean" was premature: on **nvim 0.12.2** two of josean's pinned (older) plugins were actually broken. Both traced to the same root cause — his `lazy-lock.json` pins versions from an earlier Neovim era, and this machine runs bleeding-edge 0.12.2. Fixed by updating the plugins to current and pinning treesitter to its stable branch.

## Symptoms
- **nvim-tree:** `o`/`<CR>` didn't open folders/files — `E21: Cannot make changes, 'modifiable' is off`. The tree rendered but its default keymaps never attached (old nvim-tree vs the 0.12 API), so the keys fell through to native vim.
- **nvim-treesitter:** after the fix below, `:Lazy update` moved treesitter onto its new **`main`** branch (a rewrite that deletes the `nvim-treesitter.configs` module). josean's `treesitter.lua` + textobjects use the old `.configs` API → `Failed to run config for nvim-treesitter … module 'nvim-treesitter.configs' not found`.

## Fixes
- `:Lazy update` — bumped all plugins to current versions; this alone fixed nvim-tree's keymaps (`o`/Enter now open nodes).
- `nvim/lua/grim/plugins/treesitter.lua` + `nvim-treesitter-text-objects.lua` — added `branch = "master"` so updates stay on the stable API branch (master is still maintained and keeps `.configs`).
- `nvim/lazy-lock.json` — stripped the stale `main`-branch treesitter/textobjects entries, deleted the on-disk clones (`~/.local/share/nvim/lazy/nvim-treesitter{,-textobjects}`, runtime state — regenerated), so lazy re-cloned both fresh on `master`. `:TSUpdate` rebuilt the parsers (20/20).

## Result
nvim-tree opens with `o`/`<CR>`; treesitter loads clean, parsers installed. The config is now **off josean's exact pinned versions and on current ones** — unavoidable on 0.12.2.

- If other pinned plugins misbehave on 0.12.2, the pattern is the same: `:Lazy update` it, and pin `branch = "master"` for any plugin whose config uses an API its `main`/default branch has dropped.
